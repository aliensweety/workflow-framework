# decisions.md

本项目的自述文档:是什么、要成为什么、哲学边界、长期取舍、历史决策。

**平时不读。** 只在以下情况打开:
- 做架构决策时
- 回头反思设计时
- 判断"这次改动是否真的让项目变好"时
- 自我迭代时

不在运行时加载——它不是给 Claude Code 执行用的,是给思考用的。

---

## 心法(不随项目变,每个新项目继承)

### 是什么 / 要成为什么

每次回头看项目,先问两件事:

1. **当前这个项目是什么?**
2. **它要成为什么?**

这两个问题明确了,对自我的认识才能清晰。对的方向、好的方向,是自我迭代的关键基石。

### 每次改动的双向检查

**正向**:这次改动是否真的让项目变好?还是只是我们觉得它变好?

**反向**:这次改动是否让项目更难变好?
- 引入了未来想改也改不动的字段?
- 把可选变成强制?
- 把简单变成复杂?
- 让 Claude Code 多了一次本不必要的判断?

两个一起问。正向查收益,反向查成本。

### Anthropic 的提醒

> 不要告诉 Claude 什么不应该做,而是要告诉他该做什么,相信模型。

每次想加规则、加限制之前回到这条。规则越多,AI 反而越保守、越事事询问。

### 自主判断不是免费的

"让 Claude Code 自主"听起来优雅,但每一次自主都意味着:
- 重新读上下文做判断 → 消耗注意力
- 不同会话可能得到不同答案 → 不稳定性
- 容易被略过(没有触发信号就不会自发发生)

判断结果**会变化**的事 → 让它自主;判断结果**永远相同**的事 → 写死成字段。

## 历史决策

> 重大决定记在这里。每一条包括:什么时间、为什么这么决定、放弃过什么方案、有什么前提。

### 示例格式

```
2026-05-20  把 X 从必填字段降为可选
原因:发现 80% 的工作流都不需要。
放弃方案:保留必填+用默认值兜底(增加了配置噪音)。
前提:如果未来发现"忘了写 X 导致出错"的频率超过 N,重新审视。
```
