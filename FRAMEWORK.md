# FRAMEWORK

这份文档是整个框架的 DNA——设计指向、核心价值、关键机制、边界。

**这份文档不在运行时默认加载**。它放在框架根目录（不在 `install/` 下），是给人读的、给 Claude Code 按需读的设计文档，不是每次运行都要消耗的注意力。

当你对"为什么这么设计"有疑问，或者准备给框架加字段、加规则、加机制之前，读这份。

---

## 这个框架是什么

一个 Claude Code 驱动的工作流框架。让"稳定可重复的生产流程"能被 Claude Code 用自然语言搭建、运行、修改。

它在光谱上的位置：N8N 的可重复执行能力 + Claude Code 的自然语言灵活性。前者是 N8N 的强项，后者是 N8N 的死穴。

## 为什么非要这个框架

Claude Code 用 skill 或纯对话就能"编排步骤"。这个框架真正的价值不是编排——**是把每次执行变成可持久化、可观察、可局部修改的实例**。

`manifest.yaml` + `runs/` 目录让这些事成为可能：

- 跑到一半中断了，下次 resume 继续
- 跑完一周后回来说"第 3 段改一下"，Claude Code 能定位并只重跑受影响部分
- 同一个流程跑过 10 次，每次的产出和中间态都在 `runs/` 里留着，可对比可回溯
- 同一份 workflow 模板可以对不同选题重复生产（模板 vs 实例分离）

如果你只是一次性跑完就扔，用 skill 或直接对话更简单。需要上面这些能力时，才用这个框架。

## 方向 / 长期取舍

- **偏向**：自然语言一等公民 · 实例可观察 · 局部可改
- **取舍掉**：通用性（现阶段为内容生产打磨，不前置设计跨领域迁移）· 一次设计完美 · 跨 workflow 编排
- **不为用户偏好转向**：核心定位是给"重复性生产"提供持久化和局部修改能力。偏离这个的请求，拒绝。

## 三条不可妥协的核心价值

### 1. Claude Code 是主语，不是被规范的对象

框架存在是为了**让 Claude Code 跑得更顺**，不是为了**约束 Claude Code 的行为**。

每加一条规则前问一句："Claude Code 自己能做到吗?"——能，就不要写。

正面表现：
- 不规定脚本的输出 JSON 结构（每个脚本输出形态本来就不同，Claude Code 看实际输出再决定怎么承接）
- 不靠固定字段名定位 unit（让 Claude Code 读 manifest 整体识别）
- 不预设所有可能的边界情况（遇到了再加）

反例（要警惕的）：之前一个 cc 工作流项目"事事询问"——根因就是规则越加越多，每个节点都成了潜在 checkpoint，Claude Code 不知道哪些该问哪些不该问，保守起见全问。

### 2. 自然语言是一等公民，intent 是种子

整个工作流的灵魂是 `intent` 字段。

**intent 是种子**。workflow 的 intent 是整个流程的种子；每个 step 的 intent 是该步骤的种子。

种子的特征：
- 有生长方向（目标）
- 与环境互动才能长成特定形态（可用的 skill、脚本、API 是环境）
- 同一颗种子在不同环境会长成不同样子

工作流的其他字段（method、command、consumes、produces 等）是**种子在特定环境下成长出来的具体形态**——是"这么跑被验证过可以"的记录，不是机械指令。打磨阶段可能每次都重新生成，稳定后沉淀下来减少重复判断。环境变了（脚本路径改了、API 换了），Claude Code 应该按 intent 重新判断怎么落地，而不是盲跟 command。

这决定了：
- workflow 头部 `intent` 必填，要有密度
- 每个 step 也有 `intent`，说"为什么有这一步"，不是"怎么做"
- 修改流程首选用自然语言对话，不是改 YAML
- 跨会话衔接靠 intent 重建上下文

### 3. 程序的归程序，思考的归思考

能用脚本确定性完成的事，不要拿到 Claude Code 的注意力里消耗 token。能用自然语言指引的事，不要硬写成代码。

三种节点类型对应三种 method：
- **思考型**（写文案、写 prompt、做美学判断）→ `method: skill:<name>` 或 `method: claude-code` + `autonomy: ask`
- **执行型**（调 API、跑脚本、确定性转换）→ `method: script` + `autonomy: auto`
- **衔接型**（抽字段、切段、装配）→ 打磨期 `method: claude-code`，稳定后下沉为 `method: script`

判断标准：**这件事重复跑会不会得到一样的结果?** 会 → 脚本；不会 → 思考。

## 关键机制

围绕上面三条核心价值，框架提供这些机制——每一个都是为解决一个具体问题。

### 机制一：autonomy 三级

`auto` / `report` / `ask`。

- `auto`：静默执行，失败才停
- `report`：执行完一句话简报，继续
- `ask`：执行完停下，展示产出等确认

打磨期把思考型节点设 `ask`，稳定后批量降到 `report`。这是"打磨→稳定"两阶段切换的唯一旋钮。

### 机制二：模板与实例分离

- `workflow.yaml` 是模板，不变。
- `runs/<timestamp>_<instance>/` 是一次具体生产。
- `manifest.yaml` 是这次运行的状态文件。

模板描述"一类生产怎么做"，实例记录"这一次具体做了什么"。修改一次实例不影响模板，也不影响其他实例。

### 机制三：units 与局部修改

units 是 framework 暴露 fan-out 结构的方式。"一篇文案 → N 张图 → N 段配音 → 合成"——把 N 个 segment 写进 `manifest.units`，framework 就能：

- 对每个 unit 跑同一个 step（`iterates_over: units`）
- 在 unit 上记录每段的产出和状态
- **修改时定位到具体 unit，沿 step 链只重跑受影响 unit，其他不动**

这第三条是 units 存在的核心理由。如果 fan-out 在脚本内部解决（脚本自己循环、产出多个文件），framework 看不见结构，也就无法做局部修改。

**何时用 units**：你需要"第 3 张图改一下"这种局部重跑能力时。
**何时不用 units**：纯线性流程，或者修改总是从头重来。

### 机制四：dispatch（执行派发方式）

step 怎么被派发执行：

- `inline`（默认）：在主会话里直接跑
- `subagent`：派给 sub-agent 跑，主会话收回结果

什么样的 step 适合 subagent：
- 长时间运行（轮询、等待外部任务）
- 输出量大但只需要最终结果（research 报告、log 分析）
- 会污染主会话上下文（大量 stdout、长文档）

打磨阶段不写 dispatch（默认 inline）。打磨完发现某步符合上面特征，加上 `dispatch: subagent` 沉淀下来。**这是"验证过的形态"沉淀下来的典型例子**——一次决定，每次复用，不重复判断。

### 机制五：自然语言修正

跑完后用对话修改产出——"2:30 那段改一下"、"第 3 场的图不对"。

Claude Code 自己定位到 unit、自己沿 workflow 的 step 顺序推断下游需要重跑的范围、自己更新 manifest、自己重跑。这是和 N8N 最大的区别——N8N 要人去点节点、拖连线、改数据映射，这里只需要说话。

### 机制六：issues 反馈回路

项目跑起来后，**框架/流程的问题提交到框架仓的 GitHub Issues**：

1. 打磨期间遇到卡点、不顺 → 随手记到 `process-findings.md`
2. session 末和用户一起过一遍，决定哪些提 issue
3. 提交时用 `gh issue create -R <框架仓>`，标题带 `[项目名]` 前缀

外部 skill / 工具的问题由每个 skill 文档自带的反馈机制处理，不在这里。

## 一组明确的"不做什么"

写在这里防止以后想多：

- **不做** workflow YAML 内的复杂逻辑（条件分支、循环、错误处理）——这些是程序的事，进脚本
- **现阶段不做** 跨 workflow 共享 state。要串多个 workflow，先用文件系统手动衔接（workflow B 的输入就是 workflow A 产出目录里的某个文件）。等积累到"反复这么做很烦"再抽机制。
- **现阶段不做** 跨 step 并行调度。当前 step 之间永远串行执行。需要的话，先用 dispatch: subagent 让单步在后台跑，主会话继续后续步骤——这能解决大部分场景。等真的需要 DAG 调度（多个 step 同时启动等同步点）再设计。
- **不做** agent 间通信和 agent team 编排。dispatch: subagent 是单向派发：主会话委派任务、sub-agent 干活、回结果。不搞 sub-agent 之间互相说话。
- **不做** 全自动 pipeline（Claude Code 全程在场，是设计选择不是过渡态）
- **不做** 规范脚本的输出格式（每个脚本输出什么是它自己的事；承接靠流程）
- **不做** 前置定义所有 edge case（遇到再加）
- **不做** 为了通用性而通用——现阶段为内容生产场景打磨好；其他领域的迁移靠实战外推，不靠前置设计

## 检验新变更的三条问

每次想加字段、加规则、加机制之前，对照骨架问：

1. Claude Code 自己做不到吗?（做得到，不加）
2. 没有这个，自然语言表达就受损吗?（不受损，不加）
3. 程序和思考的边界清楚吗?（模糊就先用 claude-code 跑一阵看清楚再决定）

三个都过，才加。

## 与脚本的关系

框架不规范脚本。脚本是项目方（或 skill 维护者）自己的事。

但一个好脚本规范会让流程更顺——比如约定好 `--output` 参数的语义、文件命名规则、错误码——这些可以写进你自己管理 skill 的项目文档里，作为你的脚本的内部规范。框架本身只要求：script method 的 command 字段能调起脚本、跑完 exit code 0 表示成功。

## 演化哲学

**实践与迭代交替。** 不追求一次设计完美。

第一版跑起来，遇到具体问题，讨论根因，改。改完再跑，再遇到问题。每次改动前对照骨架的三条问自查。

以 AI 时代的落地速度，重构比修改更快。骨架是用来在每次迭代时防止方向走偏的，不是一劳永逸的蓝图。
