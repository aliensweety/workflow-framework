# FRAMEWORK

整个框架的 DNA——核心理念、关键机制、边界、演化哲学。

**不在运行时默认加载**。给人读、给 Claude Code 在做架构决策前按需读。

每次想给框架加字段、加规则、加机制之前，先读这份。

---

## 这个框架是什么

一个 Claude Code 驱动的工作流框架。让"稳定可重复的生产流程"能被 Claude Code 用自然语言搭建、运行、修改。

它在光谱上的位置：N8N 的可重复执行能力 + Claude Code 的自然语言灵活性。前者是 N8N 的强项，后者是 N8N 的死穴。

## 为什么非要这个框架

Claude Code 用 skill 或纯对话就能"做一连串事情"。这个框架真正的价值不是把步骤串起来——**是把每次执行变成可持久化、可观察、可局部修改的实例**。

具体说，框架要解决这些问题：

- 跑到一半中断了，下次 resume 继续
- 跑完一周后回来说"第 3 段改一下"，能定位并只重跑受影响部分
- 同一个流程跑过 10 次，每次的产出和中间态都在 `runs/` 里留着，可对比可回溯
- 同一份模板可以对不同选题重复生产（模板和实例分离）

不需要这些能力的场景（一次性、跑完即扔），用 skill 或直接对话更简单。

## 核心隐喻：大表 / 小表 / 笔记

理解这套架构只需要三个概念，互相对应：

| 概念 | 文件 | 角色 |
|---|---|---|
| 大表 | `workflow.yaml` | 流程定义。该做什么、按什么顺序。**不变。** |
| 小表 | `runs/<run_id>/manifest.yaml` | 进度记录。这次做到哪了、哪些 unit 完成了。**每跑完一份产出立刻更新。** |
| 笔记 | `workflows/notes/<step_id>.md` | 单步的扩展信息（出错处理、业务详述、经验坑、历史决策）。**按需创建，平时不读。** |

所有其他术语——units、fan-out、iterates_over、stale 传播——都是这三件东西的具体形态或衍生。**不要把它们当独立机制理解，永远回到大表/小表/笔记的视角。**

## 核心原则：落地是真相，种子是兜底

这是整个框架的总开关。理解了它，其他一切都顺。

**每个 step 暴露两层信息：**

- **种子（`description` 字段）**：业务逻辑——这一步做什么、从哪取输入、产出什么、为什么存在。永远暴露。
- **落地（`command` 字段）**：确定性的执行指令，带占位符。可选。

**运行时行为：**

- 有 `command` → 直接按 command 跑。不判断、不思考、不解释。
- 没 `command`，或 command 跑不通 → 回到 `description` 重新理解，生成执行方式。
- step 失败、或编排时想改、或回溯历史决策 → 打开 `notes` sidecar 文件读扩展信息。

**为什么这样设计：**

判断本身就是不稳定性来源。能写死的就写死——写死的内容是稳定执行真相，每次跑都得到一样结果。Claude Code 的自主判断能力是**异常路径**才介入的，不是正常路径的默认行为。

**颗粒度原则**：编排时越具体越好。一个 step 是脚本 + 占位符能搞定的，就只写脚本 + 占位符。description 写得清晰是为了**出错时还有路可走**，不是为了正常路径反复重判。

## 核心机制：渐进披露作为架构哲学

Claude Code 自己的 skill 系统就是渐进披露的——frontmatter 永远加载，SKILL.md 触发时加载，bundled 资源按需加载。这个框架沿用同一架构。

**三类信息按"是否进入正常路径上下文"分层：**

| 层 | 内容 | 加载时机 |
|---|---|---|
| 常驻 | 大表（workflow.yaml） | run 启动时加载，正常路径全程在上下文里 |
| 实例态 | 小表（manifest.yaml） | run 启动时加载，每跑完一份产出更新 |
| 按需 | sidecar 笔记（notes/*.md） | 只在触发条件下用 Read 工具打开 |

**sidecar 触发条件**（三选一）：
- command 跑不通
- 没有 command、需要从 description 生成执行方式
- 编排修改时想看这一步的历史决策

**为什么必须是独立文件，不是字段的"约定不读"：**

LLM 的注意力机制不存在"读进上下文但不看"。任何写进 workflow.yaml 的内容，模型每一轮都会处理。所谓"约定不读"在原理上不成立——只有**文件没被 Read 工具打开**，内容才真的没进上下文。

这一点决定了 sidecar 必须走文件系统，不能走字段嵌套。

## 大表的最终字段清单

```yaml
name: <string>                    # workflow 唯一标识
description: |                    # 整个工作流的种子
  目的、核心价值、风格约束。给未来会话的记忆锚点。

manifest_template:                # 可选，编排时声明小表骨架
  unit_type: <string>             # units 在本工作流叫什么（segment/scene/chapter）
  unit_fields: [<list>]           # 每个 unit 承载哪些字段

steps:
  - id: <string>                  # 必填
    description: |                # 必填，本步骤的种子
      业务逻辑——做什么、从哪取输入、产出什么、为什么存在
    command: <string>             # 可选，确定性执行指令（带占位符）
    consumes: <string>            # 可选，输入路径或字段
    produces: <string>            # 可选，输出（单份 / 多份 / unit 字段）
    notes: <path>                 # 可选，指向 workflows/notes/<step_id>.md
```

**6 个字段，2 个必填**（`id` + `description`），其他可选。

**变量占位符**（在 command / consumes / produces 里可用）：
- `{run.dir}` — 当前 run 的目录
- `{run.id}` — 当前 run id
- `{unit.id}` / `{unit.<field>}` — 当前 unit 的 id 或字段（只在产出多份的 step 里可用）

**取消的字段**（从旧版删除）：`method`、`autonomy`、`dispatch`、`iterates_over`、`parallelism`、step 级 `intent`、`manifest` 段。
理由：要么被 description+command 吸收，要么靠 Claude Code 在 run 时按 step 性质自主判断（如长任务自动派 subagent），要么从 produces 的形态可推导（如多份产出 = fan-out）。

## 小表的形态：单产出与多产出自适应

manifest.yaml 不需要预先选"模式"。它根据每一步实际产出的形态自然展开：

- 某一步产出**一份**东西 → `steps_status` 里就一行
- 某一步产出**多份**东西 → `steps_status` 里一行 + `units` 数组展开成 N 行

```yaml
run_id: 2026-05-19_异乡人
status: in_progress
current_step: generate_images

steps_status:
  write_script: done
  segment: done                 # 这一步把文本切成 N 段，写入 manifest.units
  generate_images: running      # 正在为每个 unit 生图

units:                          # segment 这一步产出多份，所以 units 展开
  - id: seg_001
    status: done
    text: "..."
    image: runs/.../seg_001.png
  - id: seg_002
    status: pending
```

**fan-out 不是开关，是产出形态的自然结果。** 编排时通过 `manifest_template` 声明 unit 结构，run 不必判断。

**stale 传播规则**（用于局部重跑）：用户说"seg_003 的 text 改了"——把 seg_003 标 stale，把"产出 text 之后所有 iterates_over=units 的 step"对 seg_003 标 stale，其他 unit 不动。Claude Code 自己读大表推断哪些 step 依赖被改字段，不需要声明依赖图。

## 取消的旧概念（每一项的理由）

| 旧概念 | 取消理由 |
|---|---|
| `autonomy: auto/report/ask` 三档 | 不需要预先声明节奏。**只有两种状态：正常路径继续 / 出错停下**。汇报需不需要由 Claude Code 看情况决定。 |
| `method: claude-code/skill/script` 三种 | 三种之间有大量重叠（skill 内部用脚本、claude-code 内部用 skill）。怎么做就在 description 里写、命令在 command 里写，类型标签是抽象泄漏。 |
| `dispatch: inline/subagent` | 这是执行细节，不该让用户决定。Claude Code 看 step 性质（长轮询？大输出？）自己决定派不派 subagent。 |
| `iterates_over: units` | 从 produces 写"多份/unit 字段"就能识别，不需要独立字段。 |
| `parallelism: N` | 由 Claude Code 在 run 时根据脚本能力和资源约束自主判断；写不写都不影响正确性。 |
| "打磨期 / 稳定期" 二阶段叙事 | 这只是"遇到问题就解决"的循环描述。把它命名成"期"反而暗示一个不存在的阶段切换。 |
| compose 的"采访分轮"结构 | 自由对话直到能写大表。该停就停、该问就问，不分轮。 |
| compose 问"用不用 units / 要不要局部修改能力" | 抽象泄漏——把机制内部决定暴露成用户决定。用户描述输入输出形态就够了，units 由编排器自动判断。 |

## 三条不可妥协的核心价值

### 1. Claude Code 是主语，不是被规范的对象

框架存在是为了**让 Claude Code 跑得更顺**，不是为了**约束 Claude Code 的行为**。

每加一条规则前问："Claude Code 自己能做到吗?"——能，就不要写。

反例（要警惕的）：之前一个 cc 工作流项目"事事询问"——根因就是规则越加越多，每个节点都成了潜在 checkpoint，Claude Code 不知道哪些该问哪些不该问，保守起见全问。

### 2. 落地优先于自然语言，种子兜底

正常路径上，**写死的指令是真相**。自然语言的 description 是种子，平时帮 Claude Code 理解流程意义，异常时回去重生成落地。

这一条是对"自然语言一等公民"的修正——自然语言确实是一等公民，但**在异常路径和编排路径上**。正常执行路径上，确定性指令优先。

### 3. 程序的归程序，思考的归思考

能用脚本确定性完成的事，不要拿到 Claude Code 的注意力里消耗 token。能用自然语言指引的事，不要硬写成代码。

判断标准：**这件事重复跑会不会得到一样的结果?** 会 → 脚本（写进 command）；不会 → 思考（留在 description）。

## 文档角色

每个工作流项目里有两份不同性质的 md：

### CLAUDE.md = 指导文档
- Claude Code 每次启动会读
- 简洁、执行性强、"该做什么"
- 内容：项目结构、入口点、反馈通路、工作流的当前状态指针

### decisions.md = 自述文档 + 心法
- 平时不读，只在迭代、回头反思、判断"这次改动是否真的让项目变好"时读
- 内容：项目是什么、要成为什么、哲学边界、长期取舍、历史决策、为什么放弃过 X 方案
- 顶部置心法（每个新项目从模板继承）

两者必须分开。塞在一起会互相污染：指导文档变啰嗦，自述文档变工具化。

## 反馈通路

项目跑起来后，**框架/流程的问题提交到框架仓的 GitHub Issues**：

1. 打磨期间遇到卡点、不顺、模糊 → 随手记到 `process-findings.md`
2. session 末和用户一起过一遍，决定哪些提 issue
3. 提交时用 `gh issue create -R aliensweety/workflow-framework`，标题带 `[项目名]` 前缀

**外部 skill / 工具的问题由每个 skill 文档自带的反馈机制处理**，不在这里。框架仓只收"工作流框架本身"和"项目模板"相关的反馈。

## 旧项目不回头同步

框架演化只惠及**下一个**新项目。已经存在的项目在它创建那一刻的快照上独立运行，不被回头改动。

这是有意的设计选择：
- 旧项目已经在旧机制上跑通了。强行同步意味着每次框架变更触发一轮重构。
- 如果旧项目业务逻辑真的需要新框架能力，更快的做法是**新建一个项目，参照旧项目的业务结构重新搭一遍**。
- 重构比修改快，是演化哲学的延伸。

## 一组明确的"不做什么"

写在这里防止以后想多：

- **不做** workflow YAML 内的复杂逻辑（条件分支、循环、错误处理）——这些是程序的事，进脚本或 notes sidecar
- **不做** 跨 workflow 共享 state。要串多个 workflow 就用文件系统手动衔接。等积累到"反复这么做很烦"再抽机制
- **不做** 跨 step 并行调度。当前 step 之间永远串行。如有长任务，Claude Code 在 run 时自动派 subagent 隔离上下文，但仍是阻塞等待
- **不做** agent 间通信和 agent team 编排。subagent 是单向派发，不让 sub-agent 之间互相说话
- **不做** 全自动 pipeline。Claude Code 全程在场是设计选择，不是过渡态
- **不做** 规范脚本的输出格式。每个脚本输出什么是它自己的事
- **不做** 嵌套 fan-out（两层 units）。需要两层结构时，拍平为一层，章索引作为 unit 的一个字段
- **不做** 前置定义所有 edge case
- **不做** 为了通用性而通用——现阶段为内容生产场景打磨好

## Backlog（看到了但现在不做）

- **Hooks / 程序化触发**：让工作流项目自身有意识，不全靠 Claude Code 判断。现在不知道怎么落地，留着
- **嵌套 fan-out**：5 章 × N 节这种两层结构，现阶段拍平处理
- **跨机器移植性**：当前 sidecar 文件用相对路径引用，移植应该没问题；如果哪天发现路径处理脆弱再加机制
- **多模板版本**：当一个模板打不下所有工作流类型时再分化；用 git 分支或目录复制都行

## 演化哲学

**实践与迭代交替。** 不追求一次设计完美。

第一版跑起来，遇到具体问题，讨论根因，改。改完再跑，再遇到问题。

以 AI 时代的落地速度，**重构比修改更快**。骨架是用来在每次迭代时防止方向走偏的，不是一劳永逸的蓝图。

---

## 检验新变更的三条问

每次想加字段、加规则、加机制之前，对照问：

1. **Claude Code 自己做不到吗?**（做得到 → 不加）
2. **没有这个，自然语言表达就受损吗?**（不受损 → 不加）
3. **程序和思考的边界清楚吗?**（模糊 → 先用 description 跑一阵看清楚再决定要不要进 command）

三个都过，才加。

## 反向检查（配对使用）

每次加完想加的东西，再问一遍：

> **这次改动是否让项目更难变好？**

具体追问：
- 引入了未来想改也改不动的字段?
- 把可选变成强制?
- 把简单变成复杂?
- 让 Claude Code 多了一次本不必要的判断?

正向检查"是否真的让项目变好"和反向检查配对使用。一个查正向收益，一个查反向成本。
